# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo actually is

The `package.json` name (`vibe-coding-agent`) and `README.md` describe a generic Vercel "AI sandbox" template. That's the repo's origin, not its current product. The live application built here is **RODAID**: a bicycle registration / anti-theft civic-tech platform for Mendoza, Argentina (`rodaid.net`), used by citizens, partner workshops ("aliados"), and government entities (Ministerio de Seguridad, MPF Mendoza, several municipalities) as a multi-tenant system.

The original AI-sandbox scaffold (`ai/`, `app/api/chat`, `app/api/sandboxes`, `app/chat.tsx`, `app/file-explorer.tsx`, `app/preview.tsx`, `app/logs.tsx`, `app/state.ts`, `app/actions.ts`, and the `components/chat`, `components/ai-elements`, `components/panels`, `components/commands-logs`, `components/error-monitor`, `components/file-explorer`, `components/model-selector`, `components/tabs` UI) is still present in the tree but is **not wired into the live product** — `app/page.tsx` (the real homepage) only renders `components/rodaid/*`. Don't assume changes to the sandbox/chat scaffold affect anything a user sees; conversely, don't try to bolt new RODAID features onto it.

Deployment is **Netlify**, not Vercel, despite leftover Vercel-flavored deps (`@vercel/sandbox`) and README copy: see `netlify.toml`, `netlify/functions/*.mts` (scheduled/background jobs), and `netlify/edge-functions/auth-admin.ts`.

Code, identifiers, comments, and DB columns are predominantly in **Spanish** — match that convention in new code.

## Architecture

### Layering convention

- `app/api/v1/**` — the versioned, current API surface. A typical route: authenticate with `requireUser`/`requireRole`, delegate to a `src/services/*.service.ts` function, and on failure return `jsonError(error)`.
- `app/api/**` (non-`v1`) — older or special-purpose endpoints (chat/gpt/faq/legal assistants, sandboxes, admin bootstrap) that predate or sit outside the `v1` convention.
- `src/services/*.service.ts` — business logic and external integrations (MercadoPago, blockchain/BFA anchoring, PDF/PKCS7/PKCS12 signing, IoT, Strava/Garmin, notifications, etc).
- `lib/*.ts` — shared helpers. Note `lib/marketplace.ts` is the de facto cross-cutting kernel despite its name: it defines `getPool()` (the Postgres pool), the `ApiError` class, `jsonError()`, and is where many unrelated routes import auth/db primitives from — check it first when tracing how a route gets its DB client or error shape.
- `lib/auth.ts` / `lib/session.ts` — JWT-based session and role model.
- `lib/tenant.ts` — multi-tenant resolution and RLS-scoped query helpers (`withTenant`, `withBypassRLS`, `auditTenant`). Tenants are resolved by slug (`rodaid`, `ministerio_seguridad`, `mpf_mendoza`, `municipio_*`) to a UUID, then every query in that scope runs inside a transaction with `SET LOCAL app.current_tenant_id` for Postgres RLS isolation. Government-facing APIs pick the tenant from the `X-Tenant-ID` header (`getTenantFromHeader`).

### Data layer

No ORM — raw SQL via `pg`, database provisioned through `@netlify/database` (Neon). Schema lives entirely in sequentially timestamped SQL files under `netlify/database/migrations/`; read recent migrations to understand current table shape rather than assuming `src/types/database.ts` is exhaustive.

### Migrations: a new enum value and its first use need separate deploys

Postgres refuses to use a value added by `ALTER TYPE ... ADD VALUE` until that addition has committed (`unsafe use of new value ... (55P04)`). Splitting the `ADD VALUE` and its first use into two separate **migration files** is not enough by itself — Netlify's migration runner does not commit between files within the same deploy, only between separate deploys. A migration that adds an enum value and a later migration that uses it must land in two different deploys, not just two different files of the same deploy/PR.

This was confirmed empirically (2026-07-08) while building the CIT Completo marketplace/escrow state machine: `20260708000003_extend_marketplace_escrow_cit_completo.sql` (`ALTER TYPE marketplace_publicacion_estado ADD VALUE 'PUBLICADO_PENDIENTE_CERTIFICACION'`, ...) and `20260708000004_backfill_marketplace_estado_cit_completo.sql` (an `UPDATE` that sets rows to that same new value) were pushed together in one PR/deploy and failed with exactly that `55P04` error. Splitting them across two separate deploys on the same branch — deploy `6a4eb7750949730008700e93` (only `...003`, state `ready`) followed by a second, later deploy `6a4eb8ca70cab3000845f0b2` (`...004` alone, state `ready`) — confirmed the fix: same two files, same SQL, no code change, just delivered as two deploys instead of one.

Practical rule: when a migration adds an enum value that a later migration will read/write/filter on, merge the `ADD VALUE` migration to `main` on its own, confirm its production deploy is `ready`, and only then merge the migration that uses the new value.

**Two separate migration files are not the mechanism — two separate deploys are.** It's easy to do the file-splitting correctly and still ship the bug by putting both files in the same commit/PR, which still lands them in one deploy together. This happened a second time (2026-07-09, PR #32, building the Fase 6 escrow states): `20260709000001_escrow_estados_reserva_cit_completo.sql` (`ALTER TYPE escrow_transaccion_estado ADD VALUE 'RESERVA_PENDIENTE'`, ...) and `20260709000002_reindex_escrow_reserva_cit_completo.sql` (an index predicate referencing that same value) were correctly split into two files, but both were committed and pushed together — the PR's deploy failed with the identical `55P04` error (`unsafe use of new value "RESERVA_PENDIENTE"`). Before merging any migration PR, check whether it contains an `ADD VALUE` alongside anything that references the new value — if so, split the PR itself in two, not just the files inside it.

### Fase 6 (CIT Completo): reserve/pay endpoints + a money-integrity audit that found and fixed 4 real bugs (2026-07-09)

`POST /api/v1/marketplace/[id]/reservar` and `POST /api/v1/marketplace/[id]/confirmar-pago` (delegating to `iniciarReservaCitCompleto()` / `confirmarPagoCitCompleto()` in `escrow.service.ts`) close the buyer-facing side of the CIT Completo escrow flow. Two paths:
- Bici not yet certified (`PUBLICADO_PENDIENTE_CERTIFICACION`): `/reservar` charges a **sena** (funds the Taller's 20-point verification), publicación → `RESERVADO`, escrow → `RESERVA_PENDIENTE` → (webhook) `RESERVADA`. Once `aprobarInspeccionFisica()` seals the checklist (publicación → `EJECUTANDO_LOGISTICA`), `/confirmar-pago` charges the **saldo** (precio + logística), escrow → `SALDO_PENDIENTE` → (webhook) `FONDOS_RETENIDOS`.
- Bici already certified (`PUBLICADO_CERTIFICADO`, from a prior expired reservation): `/reservar` skips the sena entirely — no second verification, the Taller already got paid once — and charges the full saldo in one payment, escrow → `SALDO_PENDIENTE` directly, publicación → `EJECUTANDO_LOGISTICA` directly. `/confirmar-pago` never applies to this path (no `RESERVADA` row ever exists for it).

`webhookPago()` now branches on a `TRANSICIONES_APROBADO` map keyed by the escrow row's **current `estado`** (never on `mp_pagos.concepto`, which is purely a bookkeeping label): `DEPOSITO_PENDIENTE→FONDOS_RETENIDOS` (generic flow, unchanged), `RESERVA_PENDIENTE→RESERVADA` (fires the `escrow.verificacion_solicitada` notification to the Taller), `SALDO_PENDIENTE→FONDOS_RETENIDOS` (both origins above land here identically).

**A dedicated money-integrity audit of this whole area (same date, prompted explicitly: "no otro bug puntual, una auditoría completa") found and fixed four real bugs before this shipped** — the kind of "automatic system action decided against stale information" class of bug, checked systematically across every expiry clock, every scheduled worker, and every notification path:
1. **`reserva_vence_en` was never cleared once a transaction reached `FONDOS_RETENIDOS`.** Any CIT Completo sale taking longer than 48h end-to-end (a normal timeline, not an edge case) would get caught by the next `procesarReservasVencidas()` sweep and **downgraded from `FONDOS_RETENIDOS` back to `CANCELADA`** — a fully-paid transaction silently un-paid, publicación reopened for a different buyer. Fixed by clearing `reserva_vence_en = NULL` in the same `UPDATE` that sets `FONDOS_RETENIDOS` in `webhookPago()`'s `SALDO_PENDIENTE` branch.
2. **No reconciliation against MercadoPago before reverting an expired reservation.** A lost or late-arriving webhook meant `procesarReservasVencidas()` would cancel a reservation that MercadoPago had already actually approved, with no trace left anywhere. Fixed: the sweep now calls a new `buscarPagosPorExternalReference()` (searches MP's `/v1/payments/search` by `external_reference`, i.e. the escrow transaction id) for any row with a still-`PENDIENTE` `mp_pagos` entry, excludes payment_ids already known locally, and — if MP reports a genuinely new approval — delegates to the normal `webhookPago()` instead of reverting. If the MP lookup itself errors, the row is **not** reverted that pass (retried next run) — erring toward not cancelling over erring toward cancelling a possibly-paid reservation.
3. **A pago-aprobado-arrives-for-an-already-terminal-row event logged nothing.** Even after fix #2, a narrow (seconds-wide) race remains where MercadoPago approves a payment in the exact window between the reconciliation check and the revert's commit. `webhookPago()` now `console.error`s explicitly (with transaccionId/paymentId/monto) whenever an `approved` payment lands on a `CANCELADA`/`RESERVA_VENCIDA` row, so this can never fail 100% silently even though it can't be fully eliminated without holding a DB lock across an HTTP call (impractical).
4. **`procesarReservasVencidas()`'s SQL only filtered on the `reserva_vence_en` timestamp, not on `estado`** — unlike `procesarAutoReleases()`, whose `SELECT` filters `WHERE estado = 'EN_CAMINO'` directly, which is exactly why that function turned out NOT to share bug #1's failure mode (confirmed by tracing `abrirDisputa()` moving `EN_CAMINO→DISPUTADA` without clearing `auto_release_en` — harmless there only because the outer query already excludes non-`EN_CAMINO` rows). Added the equivalent `AND estado IN ('RESERVA_PENDIENTE', 'RESERVADA', 'SALDO_PENDIENTE')` to `procesarReservasVencidas()`'s query as defense-in-depth, so a future code path that forgets to clear the timestamp can't repeat bug #1 either.

Confirmed live in production via `getDeploy` (`6a500438bb7c68000803b3cd`, `state: ready`, `error_message: null`, `database_migrations.files: []` — this piece needed no schema changes, every column/enum it uses already existed from earlier Fase 3/6 migrations).

**Known gap, still open, documented with an explicit TODO in both `webhookPago()` and the state-machine doc comment at the top of `escrow.service.ts`:** there is no "confirm sale/delivery" closing endpoint for CIT Completo yet. A transaction that reaches `FONDOS_RETENIDOS` via `SALDO_PENDIENTE` sits there **forever** — nobody gets paid (not the vendedor, not the Taller for logística/éxito). The generic `confirmarEnvio`/`confirmarEntrega` endpoints are not a substitute: they model postal shipping by the vendedor, not the Taller-coordinated logística CIT Completo actually needs, and they would only ever pay out the vendedor (no `registrarLiquidacionAliadoFeeLogistica`/`...FeeExito` exist yet, unlike `registrarLiquidacionAliadoFeeVerificacion`, which already fires correctly at `aprobarInspeccionFisica()` seal time). Building this closing endpoint is real, scoped-out future work — not a minor detail.

### Esquema de disputas CIT Transferencia — diseño acordado 2026-07-10, actualizado 2026-07-11 (Esquema 3 y 4 nuevos) y 2026-07-12 (resoluciones de Esquemas 3/4 + mecanismo de canon), pendiente de implementar

**Nada de lo siguiente está implementado todavía**, salvo lo marcado explícitamente como ya implementado. Es la especificación completa que Federico definió el 2026-07-10 para el flujo de venta/disputa del CIT Completo ("CIT Transferencia"), ampliada el 2026-07-11 con dos esquemas nuevos (reclamo de titularidad fuera de plataforma, e impugnación de denuncia falsa) y el 2026-07-12 con la resolución de varios puntos pendientes de esos dos esquemas más un mecanismo de canon nuevo para iniciar disputas — guardada como referencia para una sesión dedicada aparte. Quedan puntos sin resolver del todo (listados al final de cada esquema), así que no es apta para implementar de punta a punta sin antes cerrar esos puntos con Federico.

**Precios vigentes (ya en producción, no son parte de lo pendiente):** CIT Express $5.100 ARS — identidad básica, circulación, 12 meses. CIT Completo/Transferencia $28.500 ARS — habilita venta en el Marketplace, vigente hasta la transferencia de dominio.

**Estructura de fees del CIT Transferencia (ya implementada):** Fee de Verificación $18.000 — 100% Taller Aliado. Fee de Logística $15.000 (ajustado 2026-07-10, antes $20.000 — equivalente a USD 1) — 100% Taller Aliado. Fee de Éxito 2% del valor de venta — 50% RODAID / 50% Taller Aliado, solo si la venta se concreta. **Total garantizado al Taller por CIT Completo sellado: $33.000** (antes $38.000).

#### Esquema 1 — Comprador ↔ Vendedor (particulares, usuarios)

**El flujo normal:** (1) el vendedor tiene un CIT Express activo (12 meses) → puede publicar gratis en el Marketplace; (2) un comprador reserva la bici y paga la seña; (3) la seña financia la verificación de 20 puntos del Taller Aliado (CIT Transferencia); (4) el taller sella la verificación → el CIT queda inmutable (no se cancela nunca, es un hecho técnico verificado); (5) el comprador confirma y paga el saldo → venta concretada.

**Si la venta NO se concreta:**

- **Caso A — comprador sin fondos / no confirma a tiempo (ya implementado, Fase 5):** pasadas 48hs desde el sellado del taller sin confirmación del comprador, la seña NO se devuelve (financió el trabajo real del taller, que ya se ejecutó); la publicación vuelve a estar disponible — como el CIT ya está sellado, el próximo comprador paga directo el saldo, sin nueva seña ni nueva verificación. **Prioridad de recompra por 3 días hábiles — RESUELTO 2026-07-11:** la publicación sigue visible en el marketplace durante los 3 días (no se oculta), pero el sistema bloquea cualquier intento de reserva de un tercero hasta que pasen los 3 días hábiles. El comprador prioritario tiene que pagar de nuevo dentro de los 3 días para confirmar — no alcanza con solo manifestar intención (sin botón de "reservar sin pagar"). Qué paga: como la bici ya tiene el CIT Completo sellado (ya se pagó en la venta original que se cayó), el comprador prioritario paga directo el valor de la bici más el embalaje/logística si todavía no se ejecutó — mismo trato que un comprador nuevo de una publicación `PUBLICADO_CERTIFICADO`, mismo camino técnico que ya calcula y cobra `iniciarReservaCitCompleto()` en su rama `PUBLICADO_CERTIFICADO` (endpoint `/reservar`, no `/confirmar-pago`) — **no hace falta un flujo de pago nuevo**. No vuelve a pagar el CIT de Transferencia (verificación ya cubierta). Implementación técnica pendiente: cómo se identifica "es el mismo comprador" en el endpoint de reserva/confirmación (¿por `comprador_id` de la transacción vencida más reciente sobre esa publicación?), y qué pasa si pasan los 3 días hábiles sin que confirme (la publicación se abre a cualquiera, sin más trámite) — `iniciarReservaCitCompleto()` hoy no distingue comprador prioritario de uno nuevo.

- **Caso B — disputa entre comprador y vendedor (nuevo, sin implementar):** el CIT nunca se cancela — es un hecho técnico, separado del resultado comercial de una venta puntual. La venta puntual sí se cancela — la publicación vuelve a `PUBLICADO_CERTIFICADO`, disponible sin nueva verificación. **Confirmado por investigación de Claude Code (2026-07-10): el mecanismo de disputa genérico existente (`abrirDisputa()`/`resolverDisputa()` en `escrow.service.ts`) NO sirve para esto tal cual** — es alcanzable por accidente desde CIT Completo (ambos flujos llegan a `FONDOS_RETENIDOS`), pero `resolverDisputa()` reactiva la publicación con `UPDATE marketplace_publicaciones SET estado = 'ACTIVA' WHERE estado = 'PAUSADA'`, nombres de estado exclusivos del flujo viejo — para una publicación CIT Completo ese UPDATE no matchea ninguna fila y no hace nada (sin error, pero sin efecto). Ninguna de las dos funciones referencia `tx.plan` en ningún lado (y de hecho `tx.plan` no sirve para distinguir CIT Completo del flujo genérico — ver la nota sobre `escrow_transacciones.plan` en la sección de transferencia de titularidad más arriba; el campo confiable es `aliado_id IS NOT NULL`). Hay que construir un mecanismo de disputa nuevo, específico para CIT Completo, no reusar el existente. Reputación del vendedor en escalones: 1ra cancelación con evidencia → AMARILLO automático, solo advertencia, sin revisión humana. 2da cancelación con evidencia → SIEMPRE dispara revisión humana (nunca automática) antes de cualquier sanción — un humano mira la evidencia de ambos casos y decide si se confirma o se descarta. Naranja confirmado por un humano → cuenta marcada como alto riesgo/posible estafador, más una deuda hacia RODAID por el costo de verificación ya ejecutado (no queda "gratis" para él) — mecanismo de cobro (descuento de próxima venta vs. bloqueo hasta saldar) a definir. Extinguir la cuenta → NUNCA automático, siempre decisión humana posterior sobre el patrón completo (no necesariamente ligada al segundo caso).

**Protección contra fraude a escala (bicicleterías/revendedores):** múltiples CIT de Transferencia sin venta concretada, por sí solo, NO es prueba de mala fe (puede ser alto volumen legítimo o mala suerte genuina) — pero SÍ prioriza la cola de revisión humana para las futuras cancelaciones con evidencia de ese vendedor. Tope objetivo a definir: cantidad de "verificaciones financiadas por seña sin venta concretada" que un mismo vendedor puede acumular en una ventana de tiempo (ej. por mes) antes de que sus publicaciones nuevas dejen de financiarse con la seña del comprador — el vendedor pasaría a pagar la verificación por adelantado hasta regularizar su historial. Objetivo: quitar el incentivo económico del fraude a escala sin necesitar demostrar colusión caso por caso. Señales adicionales de patrón de fraude coordinado (para detección, no para sanción automática): mismo comprador/grupo chico de cuentas reservando múltiples bicis de un mismo vendedor; cuentas de comprador muy nuevas sin historial; ratio de cancelaciones muy por encima de vendedores comparables.

#### Esquema 2 — Taller Aliado ↔ Comprador (vía el CIT Transferencia)

El Taller Aliado siempre cobra el 100% del Fee de Verificación ($18.000) y el 100% del Fee de Logística ($15.000) por el trabajo que efectivamente realiza — sea cual sea el resultado final de la venta (Caso A o B de arriba). Recibe además el 50% del Fee de Éxito (2% del valor de venta) SOLO si la venta se concreta. El taller nunca pierde el pago por la verificación ya ejecutada, incluso si la venta se cae — protege al taller de trabajar gratis por una disputa ajena a su trabajo técnico.

**Sanción a Talleres Aliados que intenten defraudar a la plataforma (nuevo, sin implementar):** aplica solo cuando hay evidencia presentada (mismo principio que el Esquema 1 — sin evidencia, no hay sanción). Con evidencia confirmada: se descuenta el monto correspondiente al CIT de Transferencia en cuestión, y queda registrado el antecedente en el sistema por 24 meses. Mecanismo exacto de "descuento" (de qué próxima liquidación, cómo se aplica) — pendiente de definir.

#### Esquema 3 — Reclamo de titularidad (venta fuera de la plataforma, sin `/transferir`)

Caso de negocio (confirmado 2026-07-11): un comprador adquiere una bici ya registrada en RODAID por otro dueño, pero la venta se hizo por fuera de la plataforma (sin usar el Marketplace ni `/transferir`) — el dueño anterior nunca confirmó nada, la base de datos sigue diciendo que la bici es suya, mientras el comprador la tiene físicamente en su poder. Por qué esto no puede resolverse solo con la palabra del comprador: permitir que alguien cambie `propietario_id` sin ningún tipo de confirmación crearía un vector de fraude — cualquiera podría "reclamar" una bici ajena mientras el dueño real todavía la tiene, sin enterarse.

**Diseño confirmado — mismo patrón que el sistema de disputas del Marketplace (evidencia + revisión humana):** el comprador inicia un "reclamo de titularidad" para una bici ya registrada a nombre de otra persona, sube evidencia de la compra (comprobante de pago, capturas de chat/WhatsApp con el vendedor, mail — mismo mecanismo de upload que el sistema de disputas del Marketplace), y un humano de RODAID revisa el caso antes de autorizar la transferencia forzada. Solo con esa autorización humana se ejecuta `transferirTitularidadBicicleta()` con un `motivo` nuevo (ej. `'reclamo_con_evidencia'` — **hoy el CHECK constraint de `cit_transferencias.motivo` solo acepta `'venta_marketplace'`/`'transferencia_manual'`; hace falta extenderlo cuando se construya esto**), preservando el CIT igual que los otros dos caminos.

**Ajuste crítico (confirmado 2026-07-11) — peor caso: estafa/robo que intenta "blanquearse" vía el propio mecanismo de reclamo.** Escenario de riesgo: alguien roba (o compra a sabiendas de que es robada) una bici registrada a nombre de un tercero ajeno por completo al reclamo, y usa evidencia falsa/fabricada para que un revisor humano apruebe la transferencia — sin que la víctima real se entere de que está pasando. A diferencia de las disputas del Marketplace (donde ambas partes ya interactuaron dentro de RODAID, dejando rastro), acá el dueño anterior nunca participó — el reclamo no puede resolverse escuchando una sola campana. Por eso el flujo de resolución tiene que incluir al dueño actual registrado:
1. Al iniciarse el reclamo, se notifica AUTOMÁTICAMENTE al dueño actual registrado — aviso claro de que alguien está reclamando la titularidad de su bici, con un plazo para responder.
2. Si el dueño actual responde y NIEGA la venta → el reclamo se rechaza automáticamente (sin revisión humana adicional) — es evidencia de mala fe clara del reclamante, y debe quedar como antecedente grave en su cuenta (mismo criterio de reputación del Esquema 1).
3. Si el dueño actual no responde dentro del plazo → recién ahí pasa a revisión humana con la evidencia presentada — el revisor debe ver explícitamente que hubo silencio, no confirmación, como dato relevante para su decisión.
4. Si el dueño actual CONFIRMA que sí vendió → se aprueba directo, sin revisión humana adicional (el interesado real ya lo confirmó).

**Pendiente de definir en detalle (para cuando se construya):** (1) ¿se notifica al dueño anterior antes de resolver el reclamo? — RESUELTO, ver "ajuste crítico" arriba: sí, siempre, es obligatorio. (2) ¿Cuánto tiempo se le da al dueño anterior para responder antes de escalar a revisión humana? — **RESUELTO 2026-07-12: 48 horas**, mismo plazo que ya usa el sistema en otras partes. Antes de escalar a revisión humana, el sistema promueve primero un cruce de datos automático con el Ministerio Público Fiscal — el mismo mecanismo que `ejecutarCrossReference()`/`clasificarNivelCIT` ya usa para CIT Express — es decir, el reclamo de titularidad se trata, en ese primer paso, como si el reclamante estuviera solicitando un CIT Express nuevo sobre esa bici. Solo si ese cruce no resuelve nada, pasa a revisión humana. (3) ¿Quién tiene el rol de "revisor"? — **RESUELTO 2026-07-12:** un humano de RODAID (staff/admin existente; no se define un rol dedicado nuevo por ahora).

**Prioridad: baja por ahora** — Federico decidió no construir esto todavía (etapa MVP, volumen bajo). Queda documentado como diseño confirmado para retomar cuando el volumen de este tipo de caso lo justifique.

#### Esquema 4 — Impugnación de denuncia falsa (vendedor deshonesto denuncia DESPUÉS de vender por afuera)

Caso de negocio (confirmado 2026-07-11): un vendedor publica y vende su bici por fuera de RODAID (redes sociales, sin usar `/transferir` ni el Marketplace), cobra, entrega la bici — y DESPUÉS, dentro de las 72hs, registra una denuncia de robo/hurto sobre esa misma bici en RODAID. La denuncia bloquea la bici en toda la red — el comprador real, que pagó de buena fe, queda con una bici marcada como robada sin haber hecho nada malo.

Por qué es más grave que un caso de denuncia común: el sistema de denuncias fue diseñado asumiendo buena fe del denunciante (ver Fase 7: "el silencio favorece al denunciante" para el caso de terceros). Acá el denunciante ES el propio vendedor, y la denuncia no es un error de detección — es una herramienta de estafa deliberada, usando el mecanismo de protección de RODAID en contra de un tercero inocente. Conexión directa con el Esquema 3: mismo principio de fondo — nadie debe quedar a merced de una sola versión de los hechos sin poder defenderse. El Esquema 3 protege al dueño legítimo de un reclamo falso; este Esquema 4 protege al comprador real de una denuncia falsa.

**Diseño confirmado:** un comprador cuya bici quedó bloqueada por una denuncia debe tener un mecanismo para objetarla/impugnarla — subiendo su propia evidencia de la compra (comprobante de pago, chat con el vendedor, transferencia bancaria, etc.), mismo patrón de evidencia + revisión humana que el resto del sistema. La impugnación dispara revisión humana obligatoria (mismo criterio que el Esquema 3 — nunca resolver escuchando una sola campana) del caso completo: la evidencia del denunciante original (si la hay) vs. la evidencia del comprador que impugna. Si se confirma que la denuncia fue falsa/de mala fe: se levanta el bloqueo sobre la bici, la titularidad se transfiere al comprador real (mismo mecanismo de `transferirTitularidadBicicleta()`, motivo similar al reclamo del Esquema 3), y el vendedor denunciante queda con un antecedente grave — mismo nivel de sanción que la mala fe confirmada del Esquema 1, o más grave todavía (esto es una denuncia falsa a sabiendas, no solo una cancelación de venta).

**Pendiente de definir en detalle:** (1) ¿cualquier usuario puede iniciar una impugnación, o solo alguien que demuestre algún vínculo previo con la bici? — **RESUELTO 2026-07-12:** solo alguien que demuestre algún vínculo previo con la bici. Medios de prueba aceptados, en orden de fuerza: factura de compra, recibo de escribano, fotos que muestren la posesión real del rodado, otros medios de prueba fehaciente y, en última instancia, testimonio de un testigo — aclarado explícitamente como NO determinante por sí solo, el más débil de los medios de prueba. (2) ¿Plazo para impugnar una denuncia antes de que quede firme? — **RESUELTO 2026-07-12 (corregido el mismo día): 15 días hábiles** — revisado al alza respecto a la primera respuesta (1 día hábil), para dar tiempo real a reunir evidencia seria (factura, recibo de escribano, fotos, etc.). (3) ¿Mismo "revisor humano" que el resto del sistema, o algo específico dado que acá hay una denuncia judicial de por medio (las denuncias reales pasan por el MPF) — ¿RODAID puede levantar el bloqueo por su cuenta si la denuncia judicial en sí sigue activa en el sistema del MPF, o solo puede desvincular el bloqueo INTERNO de RODAID sin tocar el estado legal real? Sigue sin resolver.

**Prioridad: baja por ahora**, misma etapa MVP — queda documentado como diseño confirmado para retomar junto con el Esquema 3 (comparten la misma infraestructura de "evidencia + revisión humana + notificación a la otra parte").

#### Archivo de jurisprudencia — pieza transversal para los 4 esquemas (confirmado 2026-07-11)

Federico pidió que cada resolución de conflicto (Esquemas 1-4, cualquier caso que pase por revisión humana) quede documentada como un "archivo de jurisprudencia" — casos resueltos, con la metodología aplicada, consultable por los propios usuarios de la plataforma. Objetivo: consistencia entre revisores (no se resuelven casos parecidos de forma arbitraria), y transparencia real hacia la comunidad sobre cómo funciona el sistema de protección.

**Visibilidad confirmada:** solo para usuarios logueados, dentro del Garaje Digital — no público sin cuenta. Reduce el riesgo de que alguien sin ningún vínculo con la plataforma use el archivo para reconocimiento del sistema antes de intentar explotarlo.

**Cuidado crítico de diseño (mismo criterio aplicado toda la sesión con el chat de soporte y las páginas públicas): transparencia sin enseñar a esquivar el sistema.**

Cada caso publicado debe mostrar: qué TIPO de conflicto fue (ej. "cancelación repetida con evidencia de mala fe del vendedor", "reclamo de titularidad por venta fuera de la plataforma"); CÓMO se resolvió, en términos generales (ej. "se confirmó la sanción", "se rechazó el reclamo por falta de evidencia suficiente"); el PRINCIPIO general aplicado (ej. "RODAID prioriza proteger al comprador cuando hay evidencia clara y el vendedor no responde en el plazo").

Cada caso NUNCA debe mostrar: nombres reales ni ningún dato que permita identificar a las partes (anonimizado — algo como "Caso #047", sin usuario/bici/fecha exacta identificable); el detalle técnico EXACTO del mecanismo de detección que disparó la revisión (ej. no decir "se detectó al llegar a la 2da cancelación en 15 días" — ese nivel de precisión es un manual de cómo esquivarlo la próxima vez); el contenido literal de la evidencia presentada por cualquiera de las partes.

**Diseño técnico (a definir en detalle cuando se construya, junto con los Esquemas 1-4):** cada vez que un revisor humano resuelve un caso (en cualquiera de los 4 esquemas), además de la resolución interna, se genera un resumen anonimizado para el archivo de jurisprudencia — probablemente un campo de texto que el propio revisor completa al resolver (no generado automáticamente, para mantener control humano sobre qué tan anonimizado y qué tan general queda el relato). Página nueva dentro del Garaje Digital (ej. `/garaje/jurisprudencia` o una sección del Garaje existente) que lista los casos, filtrable por tipo de conflicto. Esto depende de que los Esquemas 1-4 ya tengan su mecanismo de revisión humana construido — no tiene sentido construir el archivo de jurisprudencia antes que los sistemas que generan los casos. Se construye junto con o después de esos esquemas, no antes.

**Prioridad:** misma etapa que los Esquemas 1-4 — todos low priority para el MVP actual, quedan documentados juntos para una sesión de diseño/construcción dedicada más adelante.

#### Mecanismo de canon para iniciar disputas (confirmado 2026-07-12) — inspirado en la Oficina de Conciliación Civil y Comercial (OCC) de Mendoza

Federico investigó el modelo real de conciliación prejudicial de Mendoza (Ley 9.388, OCC, en funcionamiento desde febrero 2025) como inspiración para un mecanismo de canon dentro del sistema de disputas del Esquema 1 — con el objetivo de evitar el desgaste del personal de RODAID en reclamos frívolos. Diferencia importante con el modelo real: la OCC cobra una multa (medio JUS) por INASISTENCIA a la audiencia, no por iniciar el trámite, y solo cobra tasas si hay acuerdo — RODAID adapta el concepto, no la mecánica exacta.

**Diseño confirmado:** (1) el canon se cobra a quien INICIA la disputa (no a quien no coopere, a diferencia del modelo puro de la OCC) — desincentiva reclamos frívolos desde el arranque. (2) Se devuelve completo si el reclamo resulta legítimo — mismo principio de "el reclamo de buena fe no cuesta nada real" que ya rige en el Esquema 1 (mala fe) y el Esquema 3 (reclamo de titularidad); solo se pierde si se confirma que el reclamo fue infundado/de mala fe. (3) Monto: **20% del valor de la venta del rodado en disputa** — proporcional, no fijo, y sin techo máximo (confirmado explícitamente 2026-07-12, tras evaluar y descartar un techo). Se cauciona al momento de iniciar el trámite de arbitraje, por quien lo inicia.

**Por qué sin techo:** RODAID está formalizando un mercado de bicicletas que hoy opera casi completamente informal/en negro en Argentina — sin un filtro proporcional fuerte, el mecanismo de disputas podría generar una demanda de revisión (y eventualmente de intervención de profesionales del derecho) que un equipo chico no puede sostener, especialmente en operaciones de alto valor. Ajuste 2026-07-12: se reemplazó la primera propuesta ($2.500 ARS fijo) tras investigar que el sistema legal argentino no tiene un porcentaje fijo uniforme para cauciones (el CPCCN art. 199 deja el monto a discreción del juez, caso por caso, según "verosimilitud del derecho") — el 20% se tomó como referencia de un criterio real y objetivo (el Código General del Proceso de Colombia lo fija así para medidas cautelares) aplicado sobre la propia escala de precios de RODAID, más coherente con la lógica de "formalizar un mercado informal" que un monto fijo simbólico.

**Pendiente de definir para cuando se implemente:** en qué punto exacto del flujo de disputa se cobra este canon (¿al presentar la evidencia inicial, antes de que un humano revise?), y cómo se integra con el mecanismo de pago ya existente (MercadoPago, mismo patrón que el resto del sistema).

#### IA predictiva en el sistema de disputas — deliberadamente pospuesto para después del 2026-07-11

Antes de construir nada hay que definir con precisión sobre qué predeciría (¿riesgo de un vendedor antes de que pase algo? ¿lectura automática de evidencia? ¿patrón de fraude a escala?) y mantener el límite ya acordado: el naranja (sanción seria, posible extinción de cuenta) **siempre** requiere revisión humana, nunca una decisión automática de IA. Ir más allá — que la IA decida o sugiera fuertemente sobre casos individuales — es un riesgo real de perjudicar vendedores honestos y quedar indefendible ante un reclamo de trato injusto; merece su propia sesión de diseño dedicada.

### Admin View-As — a real, non-destructive way for admins to preview/test any aliado-scoped panel (2026-07-12)

Built after two rounds of manual DB patching (linking a test `aliados` row, then a real one — "Esquina Cicles" — to Federico's own `usuario_id`) turned out fragile and produced its own confusing bugs (see the `aliados.tipo` investigation above, and the wrong-Neon-project/session-mixup debugging earlier the same day). Federico asked for a systemic fix instead of another one-off patch: any admin should be able to view/test any aliado-scoped panel (today: `/taller`'s `PublicarServicioTaller`, the Panel de Inspecciones) without creating or linking any real or fake `aliados` row.

**Two modes, both read-only, both admin-gated:**
- **Vista previa** — zero DB involvement, returns static mock data (`estadoPublicacionVistaPrevia()` in `talleres-desempeno.service.ts`) so an admin with no aliado at all can see the UI populated.
- **Ver como [Aliado real]** — an admin picks a real, `aprobado` aliado from a dropdown; the panel then reads (never writes) that aliado's real data.

**The security boundary is structural, not cosmetic: only GET/read endpoints ever accept impersonation.** The shared resolver `resolverAliadoParaLectura(user, aliadoIdSolicitado)` (`src/services/inspeccion.service.ts`) is called exclusively from `GET` handlers (`app/api/v1/inspecciones/contexto/route.ts`, `app/api/inspector/cit/route.ts`, `app/api/v1/talleres/servicio-publicado/route.ts`). Every `PUT`/`POST` (mutating) handler on those same routes still uses the original strict, ownership-only resolvers (`resolverAliadoDeUsuario`/`resolverAliadoIdDelUsuario` — `WHERE usuario_id = $1`, no `aliadoId` parameter accepted at all) — an admin in "ver como" mode structurally cannot approve an inspection, report a discrepancy, or publish/edit a taller's service listing under someone else's identity, because the code path that would do that never reads the impersonation parameter in the first place. `POST /api/inspector/cit`'s handler has an inline comment spelling this out explicitly (approving/reporting a discrepancy signs a real acta; an admin in "ver como" can never attribute that action to an aliado that isn't their own).

`resolverAliadoParaLectura()`'s branches: `rol === 'aliado'` → always the caller's own aliado (`modo: 'propio'`), ignores any `aliadoIdSolicitado`. `rol === 'admin'` with an `aliadoIdSolicitado` → looks up that aliado by id (`estado = 'aprobado'` only), `modo: 'ver_como'`. `rol === 'admin'` with no `aliadoIdSolicitado` → `aliado: null, modo: 'vista_previa'`. Any other role (`inspector`) → always `modo: 'propio'`, `aliado: null` (unaffected by this feature).

**Client-side state is entirely stateless** — a single URL query param `?verComoAliado=<aliadoId>`, read by a shared hook `useVerComoAliado()` (`lib/admin-view-as.ts`, wraps `useSearchParams`). No new localStorage, no new React context. `SelectorVerComoAliado.tsx` (admin-only dropdown, fetches the already-existing `GET /api/v1/admin/aliados?estado=aprobado` — no new backend endpoint needed for the picker itself) writes to this param via `router.push`; any panel reading the hook reacts automatically. `AdminViewAsBanner.tsx` renders an unmistakable amber banner ("Vista previa..." or "Viendo como: [nombre] (Taller Aliado real) — modo solo lectura...") whenever `modo !== 'propio'`, and both wired-up panels (`PublicarServicioTaller.tsx`, `inspecciones.tsx`) additionally disable their mutating actions (publish/edit button, aprobar/discrepancia buttons) client-side when not in `'propio'` mode — a UX courtesy on top of the real server-side boundary above, not the actual security mechanism.

**Extending this to a future aliado-scoped panel** means: (1) have its `GET` handler call `resolverAliadoParaLectura(user, verComoAliado)` instead of the strict resolver, forward `?verComoAliado=` from the query string, and return `modoVista` in the response; (2) keep its `PUT`/`POST` handlers untouched (strict resolver only); (3) on the client, read `useVerComoAliado()`, append it to the `GET` fetch, and render `<SelectorVerComoAliado />` + `<AdminViewAsBanner modo={...} />`.

### Remito de Embalaje y Despacho (Fase 6b, CIT Completo) — built 2026-07-12/13

Formalizes, with a signed PDF, the packaging work order a seller gives the Taller Aliado once a CIT Completo sale's saldo is confirmed (`escrow_transacciones.estado = 'FONDOS_RETENIDOS'`), and — in a second step — serves as the Taller's signed proof that it actually packaged and dispatched the bike. Two distinct acts on one `remitos` row: `generarRemito()` (seller, explicit action — the only way the Taller finds out it needs to pack the bike) → `confirmarDespachoRemito()` (Taller, signs with its own `wallet_address`, same mechanism as the 20-point inspection acta) → triggers the Fee de Logística liquidation.

**Where it lives in the money flow — a real correction made while building this:** while investigating, found that `registrarLiquidacionAliadoFeeLogistica()`/`FeeExito()` (assumed missing per this file's own prior text and per Federico's own recollection) had actually **already been built**, in commit `233922f` (2026-07-11, the real-titularidad-transfer work) as part of `confirmarEntregaCitCompleto()` — the buyer's final "confirm receipt" endpoint, which closes `FONDOS_RETENIDOS → COMPLETADA`. Both this file's old "Known gap... no endpoint exists" text and `escrow.service.ts`'s own inline `TODO(cierre CIT Completo)` comment were stale and have been corrected. Design decision made explicitly with Federico: `registrarLiquidacionAliadoFeeLogistica()` was moved OUT of `confirmarEntregaCitCompleto()` and into `confirmarDespachoRemito()` instead — the Taller collects the embalaje fee the moment it finishes its own work (Despacho a Logística), not days later when a third party (the buyer) confirms final delivery. Same principle already documented for the Fee de Verificación (paid at checklist-seal time, regardless of whether the sale later falls through). `confirmarEntregaCitCompleto()` still liquidates the Fee de Éxito (correctly contingent on the sale actually completing).

**No new state on `escrow_transacciones.estado`/`marketplace_publicaciones.estado`:** deliberately avoided adding another enum value to either (both are real Postgres enums subject to the two-deploy `ADD VALUE` rule elsewhere in this file). `remitos.estado` (`GENERADO`/`DESPACHADO`, a brand-new enum type, safe to create+use in the same migration) is the sole source of truth for this sub-tramo, read directly by the frontend via `MiPublicacion.transaccion.remito`/`MiCompra.remito`.

**Schema:** `remitos` (correlative `numero` via a new Postgres `SEQUENCE`, `remitos_numero_seq` — the first sequence in this schema; formatted `REM-{year}-{padded}` in `remito.service.ts`, not in SQL) + the full firma-completa column set mirrored from `inspecciones_fisicas` (`firma_algoritmo`/`valor`/`certificado`/`cert_serie`/`cert_fingerprint`/`modo`, not just a hash — this file's first draft of the migration only had `firma_hash` and had to be corrected before implementing the service layer). Three new nullable `TIMESTAMPTZ` columns on `escrow_transacciones`: `saldo_confirmado_en` (stamped in the webhook's `SALDO_PENDIENTE → FONDOS_RETENIDOS` branch — deliberately NOT reusing `updated_at`, which any later write, including the reminder timestamps below, would overwrite), `remito_recordatorio_in_app_en`, `remito_recordatorio_email_en` (both still unused — belong to the reminder worker, not yet built).

**Two-deploy split, same rule as documented elsewhere in this file:** `20260712000001_create_remitos.sql` (the table/sequence/columns above, all safe) shipped separately from `20260712000002_notif_tipo_remitos.sql` (`ALTER TYPE notif_tipo ADD VALUE 'REMITO_GENERADO'/'REMITO_RECORDATORIO'/'REMITO_DESPACHADO'`) — the second migration has to merge, deploy, and confirm `ready` in production on its own before any PR referencing those values (the notification-dispatch code, still unbuilt) can merge.

**Signing:** `acta-firma.service.ts` gained a new exported `firmarCanonico(canonico: string)` — the same PKCS#12/DEV credential and Web Crypto RSASSA-PKCS1 algorithm `firmarActa()` already used, generalized to sign an arbitrary canonical string instead of only an `ActaPayload`. `firmarActa()` itself now delegates to it (additive refactor, zero behavior change for its existing callers). The Remito PDF itself is additionally signed with the Autoridad Certificadora's PKCS#7 detached signature (same `firma.service.ts`/`crearSignerRodaid()` mechanism as the CIT certificate) — two independent signature layers: one proves RODAID issued the document, the other proves a specific Taller staff member confirmed the physical work.

**Public verification (`GET /api/v1/verificar/remito/[numero]`, `/verificar/remito/[numero]`):** deliberately its own route, not a reuse of `/api/v1/verificar/[serial]` (bike identity has a different shape/veredicto than "is this document genuine") — but does reuse `verificacion.service.ts`'s IP-based rate limiter (`hashIp`/`chequearRateLimit`, fully generic, no bike-specific coupling) to share the same anti-enumeration budget. Deliberately does NOT write to `logs_verificaciones` — that table/its `tipo_busqueda` column is shaped for the bike veredicto, not a remito event; forcing it in would have meant either a schema change (not asked for) or misusing the existing values.

**Download auth gotcha, worth remembering for any future authenticated-PDF-download UI:** `GET /api/v1/remitos/[numero]/pdf` requires `Authorization: Bearer` (via `requireAuth`), so a plain `<a href>` would NOT authenticate — the browser doesn't attach that header on normal navigation. `lib/remitos.ts::descargarRemitoPdf()` fetches the blob via `authedFetch` and triggers the download from an in-memory object URL instead. (Investigated whether the existing CIT certificate download, `/api/v1/cit/[id]/certificado`, had already solved this and could be copied — it hasn't: a full-repo grep found **zero frontend callers of that route at all**, so there was no existing pattern to follow here.)

**Known gap, not fixed in this pass, deliberately deferred:** `confirmarEntregaCitCompleto()` (the buyer's final "confirm receipt" endpoint that actually closes a CIT Completo sale, referenced above) has **zero frontend callers** — confirmed via a full-repo grep for `confirmar-entrega-cit-completo` across `components/`. A CIT Completo transaction that reaches `FONDOS_RETENIDOS` and gets a Remito `DESPACHADO` has no UI path for the buyer to ever confirm final delivery, release the funds, or transfer titularidad — it would sit there indefinitely with no frontend way to close it. Same category as the other documented "backend exists, no frontend caller" gaps in this file (e.g. the `pagoPendiente`/denuncia resume-later gap). Not built as part of the Remito work — it's a separate, larger feature (a real "confirm I received my bike" UI flow) that needs its own pass.

### Datos bancarios de payout (CBU/alias) — built 2026-07-16, first real prerequisite for `ejecutarTransferencia()`

Closes the design thread opened when auditing whether CIT Completo's payout side is real (it isn't — `ejecutarTransferencia()` in `compensaciones.service.ts` is still a confirmed stub that fakes a reference string and never moves real money). Federico decided against migrating to MercadoPago Split Payments (it splits instantly at payment time, which conflicts with RODAID's escrow-holding model) and instead to keep the current single-collector-account model, paying out via direct CBU/alias bank transfer to a stored destination — this piece builds the storage/validation/gating side of that, not the transfer call itself (still a separate, larger piece, `ejecutarTransferencia()` is untouched).

**Schema:** new generic table `datos_bancarios_payout` (`beneficiario_tipo` `'usuario'`/`'aliado'` + `beneficiario_id`, `cbu`, `alias`, `titular_declarado NOT NULL`, `CHECK (cbu IS NOT NULL OR alias IS NOT NULL)`, unique index on `(beneficiario_tipo, beneficiario_id)` — one live row per beneficiary, not a history) — generic because both an individual seller (`usuario`) and a Taller Aliado (`aliado`) collect through this same mechanism (Fee de Verificación/Logística/Éxito, see `pagos_liquidaciones`). Three new nullable columns on `pagos_liquidaciones` (`cbu_destino`/`alias_destino`/`titular_destino`) hold a **frozen copy** of the destination actually used at the moment each liquidación is created — deliberately not a live join to `datos_bancarios_payout`, so that a beneficiary changing their CBU later never rewrites the record of what a specific already-issued liquidación was paid to.

**CBU validation is real, not just a length check** (`lib/cbu.ts::validarCBU()`): the Argentine CBU has two real check digits (BCRA algorithm, verified against an independent source before implementing — getting this wrong risks silently accepting/rejecting a real bank account in a money-movement flow). 22 digits total; digit 8 verifies digits 1-7 with weights `[7,1,3,9,7,1,3]`; digit 22 verifies digits 9-21 with weights `[3,9,7,1,3,9,7,1,3,9,7,1,3]`; formula `(10 - (suma % 10)) % 10`. `validarAlias()` alongside it is a lightweight format check only (6-20 alphanumeric/dots/hyphens) — MercadoPago/CBU aliases have no check-digit scheme to verify.

**Publish-flow gate:** `POST /api/v1/marketplace/publicar` now 409s (`DATOS_BANCARIOS_FALTANTES`) if the vendedor has no row in `datos_bancarios_payout` — a bike can no longer be listed without RODAID having a real destination to eventually pay the seller. `registrarLiquidacionVendedor()`/`registrarLiquidacionAliadoFeeVerificacion()`/`...FeeLogistica()`/`...FeeExito()` (all four, `compensaciones.service.ts`) now look up and freeze the beneficiary's current banking data into the new columns at INSERT time via a shared `obtenerDatosBancariosVigentes()` helper; `registrarRetribucionAliado()` (the separate, older generic-CIT retribution path, not CIT Completo) was deliberately left untouched — out of scope for this pass.

**Setter endpoint:** `POST /api/v1/usuario/datos-bancarios` (`guardarDatosBancarios()`, `compensaciones.service.ts`) lets the authenticated user upsert their own `cbu`/`alias`/`titular_declarado` — validates with `validarCBU()`/`validarAlias()` before writing, requires at least one of `cbu`/`alias`. Built specifically so the 409 publish-flow gate above has a real way out the moment this ships — without it, every seller would be blocked from publishing with no path to unblock themselves. No UI yet (deliberately out of scope for this pass), and no equivalent setter for `beneficiario_tipo = 'aliado'` (Talleres) — nothing currently gates a Taller-side action on having banking data on file, so that setter wasn't urgent for this pass, but will be needed before a real payout can ever reach a Taller.

**Backlog, explicitly deferred (not blocking):** a "vista de resumen por venta" — a report/screen aggregating `pagos_liquidaciones` by `origen_id`/`beneficiario_id` — pending until there's a real screen/report that needs it.

### Fixed 2026-07-16: los webhooks de CIT Express y denuncia-mpf no dejaban rastro cuando no podían resolver el pago a una fila local

### RESUELTO 2026-07-16: `ejecutarTransferencia()` era un stub — MercadoPago no tiene API de payout a terceros, así que el pago pasó a ser manual con confirmación explícita

Cerrado el hallazgo CRÍTICO documentado más arriba (2026-07-13): `ejecutarTransferencia()` (`compensaciones.service.ts`) nunca movía dinero real, solo fabricaba una referencia falsa. La función de transferir a un CBU/alias que sí existe en MercadoPago es una acción manual de una persona humana en la app/web — no hay endpoint público para que un comercio la dispare programáticamente.

**Rediseño:** `pagos_liquidaciones.estado` ganó un valor nuevo, `LISTA_PARA_PAGO` (migración aislada, `20260716000004_liquidacion_estado_lista_para_pago.sql`, mergeada y confirmada `ready` en producción por separado — misma regla de los dos deploys documentada arriba, ya que es un `ALTER TYPE ... ADD VALUE`). `ejecutarTransferencia()` se eliminó por completo. `procesarLiquidacionesPendientes()` ya no intenta ningún pago: solo mueve `PENDIENTE → LISTA_PARA_PAGO`, exponiendo el destino ya congelado (`cbu_destino`/`alias_destino`/`titular_destino`, del trabajo de `datos_bancarios_payout`). **Deliberadamente fuera de esta pasada:** la transferencia por lote bancario (un archivo/API real de pagos masivos contra la cuenta empresa de RODAID, cuando exista una) queda como evolución futura — no urgente mientras el volumen de liquidaciones sea manejable a mano. `GET /api/v1/admin/pagos/liquidaciones` (la Cola de Pagos) es admin-only a propósito, no expuesto a `aliado` — es una operación interna de finanzas, no algo que un Taller deba ver sobre otros beneficiarios.

**Known gap, preexistente, revisado y dejado tal cual a propósito:** una liquidación que queda `FALLIDA` no tiene ningún camino automático de vuelta a la Cola de Pagos — `procesarLiquidacionesPendientes()` solo barre `estado = 'PENDIENTE'`, nunca `FALLIDA` (mismo comportamiento que ya tenía el diseño anterior, no algo que cambió este PR). `resumenFinanciero()` la sigue contando como deuda viva, pero nada la reintenta automáticamente — hoy solo se destraba con un `UPDATE` manual en la base (`estado = 'PENDIENTE'` de nuevo, para que el próximo barrido la recoja). Cuando se construya un flujo real de reintentos, ahí se revisa; documentado ahora para que no sea sorpresa el día que aparezca una liquidación `FALLIDA` real.

### Security incident: demo-session allowed unauthenticated privilege escalation to admin (2026-07-08/09)

**What was found.** `POST /api/v1/auth/demo-session` had zero authentication and read `rol` straight from the request body: `rolPedido && ROLES_DEMO.has(rolPedido) ? rolPedido : 'ciclista'`, where `ROLES_DEMO` included `admin`. Any unauthenticated caller could `POST {"rol":"admin"}` and receive a real, validly-signed AccessToken with `rol: admin` — a complete bypass of the assumption that only `contactoarribaeleste@gmail.com` and `federicodegeaceo@rodaid.net` hold admin privileges. That token satisfied `requireRole('admin')` / `requireStaff(req, 'admin')` everywhere, including every route under `/api/v1/admin/*`.

**How it was found.** While building and manually verifying the session-invalidation feature (the `sesion_invalidada_desde` watermark, see below), the verification script itself needed an admin Bearer token to call the new admin endpoint — it got one by calling `demo-session` with `{"rol":"admin"}`, which is when it became apparent this worked with no login at all.

**Confirmed exposure window:** 2026-06-30 to 2026-07-09 (~10 days), **18 illegitimate admin accounts** generated (`usuarios` rows with `email LIKE 'demo-%@rodaid.test'` and `rol = 'admin'`, counted via a read-only query against the production Neon branch).

**Two fix attempts — the first shipped broken, the second is what's live:**
1. `705e839` gated the endpoint on `process.env.CONTEXT === 'production'`. This looked right (`CONTEXT` is the value Netlify's own deploy API reports, confirmed via `getDeploy` all session as `"context": "production"`) but **does not propagate to this route's runtime**: Next.js App Router routes run through `@netlify/plugin-nextjs`, a different runtime path than raw `netlify/functions/*.mts` (which do successfully read `process.env.URL`/`RODAID_ADMIN_TOKEN`). Confirmed broken by exploiting it live in production immediately after deploy `6a4efa3a5838480008e49d81` went `ready`: `POST https://rodaid.net/api/v1/auth/demo-session {"rol":"admin"}` still returned `200`, not `403` (the role itself was correctly forced to `'ciclista'` by a separate, working part of the same fix — only the production-gate layer was silently inert).
2. `67c3232` replaced the check with `req.headers.get('host')` against the site's real domains (`rodaid.net`, `www.betarodaid.net`, `rodaid.netlify.app`, `main--rodaid.netlify.app`, confirmed via `netlify api getSite`). Verified empirically before trusting it: a request to `https://rodaid.net` with a spoofed `Host` header (TLS SNI still genuinely targeting rodaid.net) got a `404` **from Netlify's edge**, never reaching application code — proving Netlify validates `Host` for routing and a request that reaches this route necessarily carried a real, registered `Host`. Confirmed fixed live in production after deploy `6a4f01eac78e7500089bf32f`: same exploit attempt returned `403 {"error":"DEMO_DESHABILITADO"}`.

**Permanent rule:** never read a role, permission, or any other authorization-affecting field from the body of an endpoint that has no authentication check. If an endpoint is meant to be unauthenticated (demo/onboarding flows, public forms), any privilege-bearing field it might otherwise accept from the caller must be hardcoded server-side instead, with no path for the request body to influence it.

**Related cleanup (2026-07-09):** while investigating the incident above, found and deleted three more unauthenticated debug/bootstrap endpoints that had no reason to still exist in a production system:
- `netlify/functions/set-admin.mts` and `app/api/set-admin/route.ts` — two duplicate, unauthenticated endpoints (one a raw Netlify Function, one a Next.js route) that ran a hardcoded `UPDATE usuarios SET rol = 'admin' WHERE lower(email) = 'federicodegeaceo@rodaid.net'` via a raw fetch to Neon's HTTP SQL API (bypassing `getPool()`/`@netlify/database` entirely, using `DATABASE_URL`/`NETLIFY_DB_URL` directly). One-time admin-bootstrap scripts that outlived their purpose — same underlying pattern as the demo-session incident (privileged DB mutation behind an endpoint with zero auth), just not exploitable for arbitrary accounts since the target email was hardcoded.
- `app/api/debug-db/route.ts` — unauthenticated endpoint that leaked `NETLIFY_DB_URL`'s parsed hostname/protocol/pathname and whether user/password were set (not the values themselves, but still internal infra detail with no auth gate).

None of the three were the cause of two `usuarios` rows disappearing from production overnight during this same investigation — that was ruled out separately (no `DELETE` targeting `usuarios` exists anywhere in this codebase's routes, services, or migrations, and Postgres FK `ON DELETE CASCADE` only cascades parent→child, never child→parent, so no other table's row deletion could have removed a `usuarios` row). All three were deleted outright rather than gated behind `requireAdmin`, since none had any ongoing purpose once the admin accounts they bootstrapped already exist.

### FIXED 2026-07-13: badge "Vencido" y botón "Renovar CIT" aparecían para cualquier CIT real recién aprobado (Express o Completo)

Reportado por Federico: la Raleigh Mojave 5.0 apareció "CIT Vencido" en el Garaje Digital minutos después de aprobarse su inspección física de 20 puntos (certificado PDF real descargado, confirmando que el CIT sí estaba activo).

**Causa raíz confirmada:** `src/services/garaje.service.ts::obtenerActivosUsuario()` calculaba `cit_activo` así:
```sql
COALESCE(c.estado = 'activo' AND c.fecha_vencimiento > NOW(), FALSE) AS cit_activo,
```
En SQL, `NULL > NOW()` es `NULL`, no `FALSE` — con `fecha_vencimiento` en `NULL`, `cit_activo` daba `FALSE` aunque `estado = 'activo'`, y `derivarEstado()` interpretaba exactamente esa combinación como `'vencido'`. Confirmado que esto afectaba al **100% de los CIT reales del sistema, no solo a la Mojave**: un grep exhaustivo de todo `src/services/` por escrituras a `fecha_vencimiento` (`fecha_vencimiento = $`/`fecha_vencimiento = NOW`) no encontró ninguna — ninguna función viva (incluida `validation.service.ts::procesarJob()`, que es la que realmente activa el CIT tras el cruce de 72hs o acelerada por `aprobarInspeccionFisica()`) fija ese campo jamás. **Efecto colateral encontrado y confirmado, mismo patrón, en otro archivo:** `app/api/v1/bicicletas/route.ts` (el endpoint de "Mi Garaje" que alimenta `BicycleSelector`/`publicar-flow.tsx`) tenía el `COALESCE` idéntico. Severidad alta, arreglado en el mismo PR.

**Fix aplicado (ambos archivos):** `fecha_vencimiento IS NULL` ahora se trata como "sin fecha fija registrada todavía" (vigente), no como vencido — solo cuenta como no-vigente si hay una fecha real y ya pasó:
```sql
COALESCE(c.estado = 'activo' AND (c.fecha_vencimiento IS NULL OR c.fecha_vencimiento > NOW()), FALSE) AS cit_activo,
```

**Además, en `garaje.service.ts` específicamente**, se agregó una distinción explícita CIT Completo vs CIT Express en `derivarEstado()`, como defensa en profundidad (no depender solo de que `fecha_vencimiento` nunca se fije hoy): un CIT Completo/Transferencia (marcado con `cits.metadata_json.inspeccionFisica.resultado === 'APROBADA'`, el mismo campo que ya usa `pdf.service.ts::inspeccionNota()` para el certificado — sellado por `aprobarInspeccionFisica()`) **jamás** puede mostrarse como `'vencido'`, sea cual sea `fecha_vencimiento` — coherente con que su vigencia real es "hasta la transferencia de dominio", sin fecha fija por diseño. `app/api/v1/bicicletas/route.ts` no necesitó este chequeo adicional: solo expone `citActivo` binario, sin ningún concepto de "vencido" en el frontend que lo consume (`bicycle-selector.tsx`/`garaje.tsx` no distinguen "vencido" de "sin verificar" en ningún punto).

**No es una columna de schema nueva.** Se confirmó explícitamente que `cits` no tiene ninguna columna `plan`/`tipo_producto`/`tipo_cit` (leído el `CREATE TABLE cits` completo en `20260618121000_create_cit_tablas.sql` y las 3 migraciones posteriores que le agregan columnas) — el mecanismo de "es CIT Completo" reutilizado acá (`metadata_json.inspeccionFisica`) es el mismo que ya usaba el sistema, no uno inventado para este fix. ### Auditoría 2026-07-16: modelo de datos CIT / Titularidad / BiciSalud

Pedido explícito por Federico: auditar que el CIT (identidad permanente del rodado) esté correctamente separado de la titularidad (que cambia en cada venta), y que el historial de BiciSalud no se rompa al cambiar de dueño. **1. **3. ¿El `transferOwnership` (BFA) se dispara con la seña o con el saldo? — confirmado: ya dispara solo en la liquidación final, no era un bug.** Rastreada la cadena completa: `transferirTitularidadBicicleta()` solo se llama desde `liberarFondos()` (`escrow.service.ts`), y `liberarFondos()` solo se llama desde `confirmarEntrega()`/`confirmarEntregaCitCompleto()`/`resolverDisputa()` (a favor del vendedor)/`procesarAutoReleases()` — los cuatro son cierres finales e irrevocables. La confirmación de la seña (`webhookPago()`, rama `RESERVA_PENDIENTE → RESERVADA`) no llama a `liberarFondos()` en ningún punto. El anclaje en BFA además se dispara recién después de que el cambio de dominio en Postgres ya commiteó. ### RLS en `usuarios`/`bicicletas` — una consulta directa "vacía" no significa datos perdidos

Encontrado 2026-07-18 verificando en producción, vía conexión directa de solo lectura (credencial `netlifydb_readonly`, sin pasar por la app), el fix del barrido de `fecha_vencimiento`. Un `SELECT COUNT(*) FROM bicicletas`/`usuarios` contra esa conexión dio **0** en ambas tablas, mientras `cits`/`aliados`/`inspecciones_fisicas` mostraban sus conteos reales — por un momento pareció un incidente real de pérdida de datos (`cits` referenciando `bicicleta_id`s que ya no existirían, algo que no debería ser alcanzable dado que `cits.bicicleta_id` no tiene `ON DELETE CASCADE`).

**No fue un incidente.** Confirmado vía `pg_class.relrowsecurity`: `usuarios` y `bicicletas` tienen Row-Level Security habilitada (`relrowsecurity = true`), con una policy tenant-scoped (`usuarios_tenant_policy`/`bicicletas_tenant_policy`) que exige `tenant_id = current_setting('app.current_tenant_id')` (o `app.bypass_rls = 'true'`) para que una fila sea visible. `cits`, `aliados`, `inspecciones_fisicas` y `marketplace_publicaciones` **no** tienen RLS — de ahí la diferencia. Una conexión que nunca setea esa variable de sesión (cualquier conexión directa vía `psql`/`pg`/consola de Neon con la credencial `netlifydb_readonly`, sin pasar por `lib/tenant.ts::withTenant()`) ve **cero filas** en las tablas con RLS, aunque tengan datos reales — no es que esas filas no existan.

**Para la próxima vez que alguien audite datos directamente contra producción** (fuera del flujo normal de la app, que sí abre la transacción con `SET LOCAL app.current_tenant_id`): un conteo en 0 sobre `usuarios`/`bicicletas` específicamente no es señal de nada raro por sí solo — hay que setear `app.current_tenant_id` al UUID del tenant correspondiente (`rodaid` es el de uso general) antes de confiar en el resultado, o usar `SET LOCAL app.bypass_rls = 'true'` si la credencial lo permite. Si en el futuro se agrega RLS a alguna otra tabla, esta misma sorpresa se va a repetir ahí también.

### Botón de Pánico "Modo Robo" — auditoría 2026-07-18 + construido el mismo día, con alcance real (mucho más chico que "3 acciones")

La idea original tenía 3 patas: (1) marcar la bici como robada al instante, (2) un PDF con foto para la policía, (3) alertar talleres cercanos por geolocalización. Antes de diseñar, se auditó qué de esto ya existía. Resultado, con evidencia:

1. **Marcar como robada al instante — estructuralmente imposible, confirmado en el código, no es una decisión de producto.** `registrarDenuncia()` (`denuncia-mpf.service.ts`) exige un PDF real ya presentado ante el MPF (valida firma `%PDF-`, hace OCR y cruza el titular contra el perfil verificado por MxM) y, aun validando, el bloqueo del CIT solo lo dispara un admin humano vía `accionDenuncia('aprobar')` (`lib/admin-panel.ts`) desde Moderación. Dos gates obligatorios, ninguno saltable — el botón no puede prometer bloqueo instantáneo.
2. **Biometría (FaceID/huella) — cero infraestructura, confirmado por grep exhaustivo** de `WebAuthn`/`navigator.credentials`/`PublicKeyCredential` en todo el repo y `package.json`: sin resultados. Sería la primera vez, de punta a punta.
3. **PDF de certificado — tenía identidad firmada pero cero fotos.** `generarCertificado()` (`pdf.service.ts`) ya incluye marca/modelo/serie/color/BFA/firma PKCS#7, pero los únicos `drawImage` del archivo eran del QR — nunca de una foto de la bici.
4. **Talleres cercanos por geolocalización — confirmado con una consulta de solo lectura en producción, corrigiendo un dato de una sesión anterior:** ya NO son 0 aliados (hay 4 `aprobado`: Bikeup/tienda, Bisi manía/tienda, Esquina Cicles/tienda, Taller de Prueba RODAID/taller interno), pero los 4 tienen `talleres.lat`/`talleres.lng` en `NULL` — nadie cargó su geocerca real todavía. Conclusión práctica sin cambios: no hay a quién alertar "cerca" de nadie, aunque la causa exacta cambió (existen destinatarios, no existe su ubicación).

**Alcance real construido el mismo día, dado lo de arriba:**

- **Atajo de un toque + paso de confirmación, sin prometer bloqueo instantáneo.** Nuevo botón "Modo Robo" (`components/rodaid/garaje-digital.tsx`, junto a "Denunciar robo" en `ActivoCard`) que abre `ModoPanicoStepUp` y, tras confirmar, dispara el mismo `DenunciaMpfModal` de siempre (ya recibía `bici: ActivoGaraje`, ya "pre-cargado" — no hizo falta ningún dato nuevo para eso). Texto honesto en dos lugares: una leyenda siempre visible bajo los botones de acción ("No bloquea tu bici al instante...") y el mismo texto repetido dentro del modal de confirmación — para que se lea sea cual sea el camino (con o sin contraseña, ver abajo).

- **"Re-verificar MFA" del Admin Dashboard investigado y descartado explícitamente, no asumido.** `verificarMfaYStepUp()`/`requireAdminPanel()` (`lib/admin-panel.ts`) están scopeados enteramente a `rol='admin'` con un TOTP ya enrolado (QR escaneado en una app autenticadora) — ningún ciclista tiene fila en `admin_perfiles` ni pasó nunca por ese enrolamiento. Reusarlo tal cual hubiera significado construir, desde cero, un enrolamiento TOTP para cada ciclista — y la primera vez que lo necesitaría sería en medio de un robo real, el peor momento para pedirle que configure un segundo factor por primera vez. En su lugar se construyó un paso de confirmación liviano pero real, reusando el único primitivo de "confirmar que sos vos" que ya existe para cuentas locales: `verifyPassword()` (`lib/auth.ts`), el mismo que ya usa `POST /api/v1/auth/cambiar-password`. Nuevo endpoint `GET`/`POST /api/v1/usuario/reautenticar`: el `GET` informa `requierePassword` (para saber si mostrar el campo), el `POST` verifica la contraseña contra la sesión propia. Las cuentas de Mendoza x Mí tienen `password_hash IS NULL` (`mxm.service.ts`: "Crear la cuenta federada (sin password: se autentica por MxM)") — para esas, `requierePassword: false` y el paso se salta solo, porque su identidad ya viene verificada por un proveedor gubernamental real. **Límite conocido y aceptado, no resuelto:** para una cuenta MxM con el teléfono ya desbloqueado, no hay ningún paso de confirmación adicional antes de llegar al formulario de denuncia (el destino sigue exigiendo el PDF real + aprobación humana, así que el peor caso es alguien abriendo el formulario pre-cargado con la identidad de otra persona, sin poder completarlo de verdad) — aceptable dado el destino real, pero vale tenerlo presente si en algún momento se construye un enrolamiento propio de 2FA para ciclistas.

- **PDF con foto incrustada, como página anexo (no en la página 1).** `generarCertificado()` (`pdf.service.ts`) ahora recibe `bici.fotoUrl` (nuevo campo en `CertificadoBici`, ya público y sin cifrar) y, si la descarga y el formato (PNG/JPEG, detectado por content-type + magic bytes) son válidos, agrega una **segunda página** ("Anexo fotográfico") con la foto a página completa y el código CIT al pie, antes de reservar el hueco de firma PKCS#7 (para que la firma cubra el documento completo). Se optó por una página nueva en vez de pelear por espacio en la columna derecha de la página 1 (ya muy ajustada entre el QR, la nota de inspección física y el sello) — más simple y sin riesgo de colisión de layout. Descarga/formato inválido nunca bloquea la emisión: solo se omite la página anexo, con `console.error` para visibilidad. Cableado en ambos certificados que ya existían (`app/api/v1/cit/[id]/certificado/route.ts`, privado; `app/api/v1/verificar/[serial]/certificado/route.ts`, público) — cualquier descarga de certificado se beneficia, no solo la que sale de Modo Robo.

- **Haversine de talleres cercanos — preparado, deliberadamente apagado, documentado en el propio código.** `buscarTalleresCercanos()` (nueva función en `src/services/aliados.service.ts`, al final del archivo) implementa la query lista (misma fórmula de Haversine que `lib/geo.ts::calcularDistanciaHaversine`, reimplementada en SQL nativo de Postgres con `acos`/`cos`/`sin`/`radians` para no traer todos los aliados a Node) pero **no la llama ningún endpoint ni componente** — motivo documentado in-line: mientras `talleres.lat`/`talleres.lng` sigan en `NULL` para los 4 aliados `aprobado` (punto 4 de la auditoría), conectarla mostraría "sin talleres cerca" siempre, sin valor real. Para activarla cuando corresponda: (1) confirmar que hay geocercas reales cargadas, (2) definir el radio de alerta por defecto (sin valor de negocio confirmado todavía), (3) conectarla a un endpoint + botón real en la UI, (4) definir el canal de notificación saliente HACIA un aliado (no existe ninguno hoy — `notificaciones` solo modela avisos hacia usuarios).

- **Biometría real (WebAuthn) — queda fuera de alcance, no construida.** Motivo: cero infraestructura existente (punto 2 de la auditoría) — sería una interacción nueva de punta a punta (registro de credencial + verificación en cada uso), un cambio de alcance mayor que el resto de esta pieza. Si en el futuro se prioriza, empezar por un enrolamiento de credencial WebAuthn por usuario (tabla nueva, algo como `credenciales_webauthn`) y considerar si conviene reemplazar o complementar el paso de contraseña de `POST /api/v1/usuario/reautenticar` con eso.

**Confirmado en producción (2026-07-18), de punta a punta:** (1) el certificado con anexo fotográfico se probó contra una bici real (`II1124279207`) — encontró y corrigió un bug real en el camino: `bicicletas.foto_url` es una ruta relativa (`/api/v1/marketplace/fotos/...`), no una URL absoluta, así que el primer intento generaba el PDF sin la página anexo (fallaba en silencio, por diseño) — fix en `absolutizarUrl()` en ambas rutas de certificado, confirmado después con el mismo serial dando 2 páginas / ~218KB. (2) Federico probó el botón "Modo Robo" en vivo: abre el formulario pre-cargado como se esperaba, y al subir un PDF genérico (sin denuncia real ante el MPF) el sistema correctamente NO dejó avanzar el reporte — confirma en producción que la promesa de "nunca bloqueo instantáneo, siempre exige PDF real" se cumple tal cual se diseñó.

### FIXED 2026-07-21: republicar una bici retirada fallaba — dos bugs distintos en cadena, encontrados uno detrás del otro

Bug reportado en producción: un vendedor "retira" su publicación (`retirarPublicacion()`, PR #139 → `marketplace_publicaciones.estado = 'CANCELADA'`, la fila no se borra) y al intentar republicar la misma bici, el sistema la rechaza con el mensaje genérico del cliente "No pudimos publicarla" — sin ninguna pista del motivo real.

**Investigación, con dos hipótesis descartadas antes de la causa real:**
- NO era el índice único de `cit_id` (`idx_mp_publicaciones_unica_activa_por_cit`) — ya es parcial, excluye `CANCELADA` igual que el chequeo de la app (nota: en el momento de escribir esto se creyó que su `WHERE` era solo `ACTIVA`/`PAUSADA`; en realidad cubre 6 estados vivos, ver la corrección más abajo en la sección del 2026-07-21 sobre `tienePublicacionActiva`).
- NO era el bug de `fecha_vencimiento` sin `COALESCE` — ya arreglado en `publicar/route.ts` por el PR #115 (commit `24abf61`, 2026-07-17).

**Bug real #1 — `marketplace_publicaciones.slug` tenía un `UNIQUE` GLOBAL, no parcial.** El slug se genera deterministicamente desde marca/modelo/año/bicicleta_id (`publicar/route.ts`) — republicar la misma bici genera el slug idéntico al de la fila ya cancelada, choca contra el `UNIQUE` de toda la tabla (a diferencia del índice de `cit_id`, que sí es parcial), Postgres tira `23505`, y `jsonError()` lo devuelve como `500 INTERNAL_ERROR` genérico. Confirmado con el dato real: el slug de la Raleigh Mojave cancelada coincidía exacto con el que generaría un reintento. **Fix:** mismo criterio que el índice de `cit_id` — el `UNIQUE` de `slug` pasó a índice único parcial, solo entre `ACTIVA`/`PAUSADA` (migración `20260721000001`). Confirmado que `slug` no se usa para ningún lookup en todo el repo (SQL/rutas/links, todo pasa por el UUID `id`), así que no hay riesgo de ambigüedad al permitir que publicaciones terminales compartan slug.

**Bug real #2, encontrado recién después de arreglar el #1 (estaba tapado detrás) — Swipe to Sell mandaba `foto_url` relativa donde el servidor exige URL absoluta.** `bicicletas.foto_url` se guarda como ruta relativa (`/api/v1/marketplace/fotos/...`, mismo formato ya documentado en `absolutizarUrl()` de los endpoints de certificado) — `publicarPorSwipe()` (`lib/swipe-to-sell.ts`) reenviaba esa ruta ya guardada tal cual dentro de `fotosUrls`, y el schema Zod de `publicar/route.ts` exige URLs absolutas (`z.string().url()`) → `400 VALIDATION_ERROR`, "Cada foto debe ser una URL valida." El formulario normal nunca pisa esto porque sube el archivo en el mismo request y la URL absoluta la resuelve el servidor recién después de la validación — Swipe to Sell es el único camino que reenvía una URL ya guardada como si fuera nueva. **Fix:** `absolutizarFotoUrl()` client-side (mismo criterio que `absolutizarUrl()` server-side, adaptado a `window.location.origin` en vez de `Request`).

**Hallazgo metodológico importante, encontrado a mitad de la verificación — Netlify DB (Neon) hace branching de base de datos POR DEPLOY, no comparte una única base entre previews/producción.** Cada deploy tiene su propio `database_branch_id` (visible en `getDeploy`) — confirmado con `listSiteDatabaseBranches`/`getSiteDatabaseBranch`, que devuelven una rama por cada PR/branch (la mayoría `archived` una vez mergeadas/cerradas). La conexión de solo lectura a nivel de sitio (`netlify api getSiteDatabase`) apunta **solo a la rama `production`** — usarla para verificar un fix en un deploy preview da un resultado incorrecto (mostró el índice viejo, como si la migración no hubiera aplicado, cuando en realidad sí había aplicado bien en la rama correcta del preview). Para verificar contra la base real de un preview específico: `netlify api listSiteDatabaseBranches` para encontrar el `branch_id` (coincide con el nombre de la rama de git), después `netlify api getSiteDatabaseBranch --data '{"site_id":"...","branch_id":"..."}'` para la connection string de esa rama puntual. Aplica también el gotcha de RLS ya documentado (`usuarios`/`bicicletas` necesitan `SET app.bypass_rls = 'true'` o `app.current_tenant_id`, si no la conexión ve 0 filas).

**Probado de punta a punta por Federico en el deploy preview, con la Raleigh Mojave real** (`CIT-II1124-530D29`) — retirada y republicada con éxito tras ambos fixes. Confirmado también vía API: `estado: "ACTIVA"`, mismo slug reutilizado sin chocar (`raleigh-mojave-5-0-2022-6eab28`), `fotosUrls` con URL ya absoluta. PR #143.

**Hallazgo, confirmado con consulta de solo lectura contra producción:** de 10 aliados resueltos, 9 eran datos de desarrollo — nombres de prueba explícitos ("Test Logo Email", "Test Logo Inline", "Taller Prueba RODAID"), o con nombre de negocio de apariencia real pero vinculados a **cuentas demo-session sin dueño real** (`demo-xxxx@rodaid.test` — "Bikeup" y "Bisi manía", pese a que el campo `aliados.email` mostraba direcciones que parecían reales; ese campo es solo el contacto de la solicitud, no necesariamente la cuenta vinculada vía `usuario_id`), o la cuenta `@rodaid.test` usada para verificar el gate de `aliados.tipo` en PR #146 ("Lucas"). Confirmado por Federico: **"la plataforma no tiene ningún taller real en producción"** — se conservó únicamente `Taller de Prueba RODAID (interno, no publicar)` (pedido explícito, "dejá solo el de mi admin para prueba"), que además resultó ser el único de los 10 con datos reales vinculados (4 `inspecciones_fisicas` + 1 `pagos_liquidaciones`).

**Cómo se ejecutó, sin tocar ningún ajuste de seguridad:** confirmado que no hay credencial de escritura contra producción disponible (`netlifydb_readonly` es la única expuesta por la API de Netlify — mismo hallazgo ya documentado en la sección de RLS más arriba) y que el toggle "Allow personal access tokens full access to the production database" no tiene ningún método equivalente en la API pública de Netlify (solo existe en el dashboard). En vez de pedirle a Federico que lo habilitara temporalmente, se armó como una **migración SQL normal** (`20260722000001_limpieza_aliados_prueba_produccion.sql`) — mismo mecanismo que cualquier otro cambio de esquema/datos del repo, sin necesitar ninguna credencial especial ni tocar ningún ajuste de seguridad. Verificado antes de escribir la migración (consultas de solo lectura) que ninguno de los 9 aliados a borrar tenía fila vinculada en `aliado_servicios`/`talleres`/`inspecciones_fisicas`/`escrow_transacciones`/`pagos_liquidaciones`/`remitos`/`solicitudes_reserva_taller`, y que las 3 cuentas de usuario detrás de ellos no tenían bicicletas, notificaciones, datos bancarios ni identidades federadas — solo `sesiones` propias (`ON DELETE CASCADE`, se limpian solas). Verificado después en el deploy preview de la migración (una copia real de producción a nivel de rama de Neon) que el resultado fue exactamente el esperado antes de mergear. PR #151.

### FIXED 2026-07-22: el Nav mostraba el menú completo de un usuario común a cualquier cuenta `rol='aliado'`

Bug reportado por Federico: un Taller Aliado logueado veía el mismo Nav completo que cualquier ciclista (Comprar/Vender/RODAID PAY/Seguridad/Verificar/Mi Garaje/Aliados/Asistente/Desarrolladores), todo junto a "Mi Taller" — sin ningún scoping por rol.

**Fuente de verdad confirmada antes de tocar nada** (pedido explícito): no existe un enum `ADMIN`/`TALLER_ALIADO`/`USUARIO` en el esquema — es `usuarios.rol` (`ciclista`/`inspector`/`admin`/`aliado`), embebido en el JWT, expuesto al cliente vía `useAuth().user.role` (`auth-context.tsx`, que ya documenta que esto es solo UX, no control de seguridad real). No hay forma de distinguir `taller` de `tienda` a este nivel — `aliados.tipo` no viaja en la sesión — así que el scoping nuevo aplica a **cualquier** `rol='aliado'`, sin importar el tipo.

**Confirmado también antes de tocar nada:** ocultar links del Nav es **100% cosmético**. Ninguna de las rutas ocultas (`/garaje`, `/asistente`, `/aliados`, `/desarrolladores`, `/verificar`, las anclas de la home) tiene gate por rol server-side — como mucho exigen que haya alguna sesión logueada. Un Taller Aliado que escriba la URL a mano sigue entrando igual; si se quiere bloqueo real es trabajo aparte, no incluido en este fix.

**Scoping deliberadamente angosto:** `inspector` (que comparte el mismo link "Mi Taller" desde antes) queda sin tocar — sigue viendo el Nav completo de siempre. El pedido fue específicamente sobre Taller Aliado, no un cambio global. `nav.tsx` gana `isTallerAliado = user?.role === 'aliado'`; cuando es `true`, tanto el nav desktop como el menú mobile muestran únicamente "Mi Taller" (mismos arrays `LINKS_SECCION`/`LINKS_APP` filtrados en los dos lugares, para que el mobile no quede desincronizado del desktop).

**Color del botón activo:** teal sólido `#1E9E96` — de 4 propuestas visuales (naranja sólido, teal sólido, outline con acento, degradé), Federico eligió teal porque ya es el color que el resto del Nav usa para "activo/verificado", sin competir con el naranja reservado a conversión ("Publicar mi bici"). PR #152.

### Mobile

`android/` and `ios/` are Capacitor shells (`capacitor.config.ts`, appId `net.rodaid.app`). `server.url` points at `https://rodaid.net` — the native apps are thin WebView wrappers loading the live deployment, not bundlers of a local static `webDir` build.

### UI

`components/ui/*` — shadcn/ui primitives (`new-york` style, see `components.json`). `components/rodaid/*` — the actual product UI (marketing site sections, dashboards, forms).
